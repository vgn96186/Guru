# TypeScript baseUrl → TS 6/7 Migration Plan

## Current State Analysis

| Item                | Value                                                               |
| ------------------- | ------------------------------------------------------------------- |
| TypeScript Version  | `~5.9.2` (pinned)                                                   |
| `baseUrl`           | `"."` in tsconfig.json:10                                           |
| `paths`             | `"@/*": ["*"]` in tsconfig.json:13                                  |
| `@/*` Usage in src/ | **ZERO imports found** — codebase uses relative imports exclusively |
| Metro/Jest Support  | `@/*` NOT resolved (confirmed per AGENTS.md)                        |

**Key Finding**: The `@/*` path mapping is configured but **completely unused**. The codebase already follows the AGENTS.md guideline to use relative imports.

---

## Migration Strategy: "Remove Unused Config"

Since `@/*` is not used, we can simply remove `baseUrl` and `paths` without any code changes. This is the cleanest path.

### Why This Works

1. No source files import from `@/*`
2. `baseUrl` only exists to support `paths` (which is unused)
3. Expo's `tsconfig.base` doesn't rely on `baseUrl`
4. No Metro/Jest config changes needed (they don't resolve `@/*` anyway)

## Alternative: If You Need Path Aliases in Future

If you later want to add path aliases back (e.g., for cleaner imports), use **subpath imports** (package.json `"imports"` field) instead of `baseUrl` + `paths`:

```json
// package.json
"imports": {
  "#src/*": "./src/*",
  "#components/*": "./src/components/*"
}
```

Then import via:

```typescript
import { Button } from '#components/Button';
```

This is the TypeScript 6.0+ recommended approach and works with ESM/bundlers.
